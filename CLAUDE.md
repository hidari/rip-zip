# CLAUDE.md

## プロジェクト概要

クロスプラットフォームなzipファイルの生成・展開コマンド rip を作るプロジェクトです。
関数型プログラミングの原則を重視した開発を行う一人開発・運用が前提のプロジェクトです。

## [MUST] 基本原則

- ユーザーデータを守るセキュリティを第一に考える
- 問題に対する解決策は局所最適を避け、**常に全体最適な方法**を選ぶ
- 一人での開発・運用に最適化するため複雑性を避け、できるだけシンプル（NOT EASY, NOT WORKAROUND）な解決策を選ぶ
- **テスタブルなコードを徹底** して単体テスト、統合テストを必ず記述しテストで全機能の動作を保証する
- コードとテストが**ファーストクラスドキュメント（source of truth）**、テストコードは**仕様としてのテスト**の考え方を徹底する
- 手動テストによる動作確認を最小限にすることをなにより重要視する
- **ボーイスカウトルールに必ず従い** コードは変更前よりもより良い状態にする
- 関数型プログラミング的アプローチを強制し、厳格なエラーハンドリングパターンと型安全を徹底して開発する:
    - 副作用を最小限に抑え、純粋関数を優先する
    - イミュータブルなデータ操作を心がける
    - 早期リターンで深いネストを避ける
    - エラーハンドリングパターンを統一することによる一貫性を重視する

## [MUST] 必ず守らなければならないルール

- 原則として feature-devプラグイン を使用する
- 作業開始前に `git-branch-switcher` スキルでブランチを選択する
- コードの修正が終わったら最初に `cargo xtask fmt` でLint等を実行する
- ブランチのpush後は以下の手順でPull Requestを確認してください
    1. `gh pr list --head $(git branch --show-current) --base main` で Pull Request の存在を確認
    2. Pull Requestが無ければ `gh pr create --assignee @me --base main --fill` で作成
- [MUST] 作業中のコマンド実行は **絶対に** `cargo xtask` の開発用コマンドを使用する（該当するサブコマンドが無ければ追加する）
- シークレット情報（APIキー、パスワード等）をコードにハードコードしない
- シークレットは環境変数経由で注入し、コンソールやログに表示されないようにする
- `.env.example` にキー名のみ記載し、実際の値は `.env`（.gitignore済み）で管理する
- ローカルからのリリースは **絶対に行わないでください**（必ずGithub Actionsを通す）
- 既存コードの変更時もユニット/統合テストは必ず書き、他のテストを参考に一貫性を持たせる
- 外部のライブラリやブラウザのAPIはそのまま使うのではなく、**Library Contractパターンを使い最小限の公開APIと契約テスト**でメンテナンス性と堅牢性を維持する
- PRのマージ前には必ず `/simplify` と `feature-dev` によるレビューを実行すること
- 警告やエラーは処理中のタスクや変更に関係ないものでも即座に修正し、常に警告0個を維持すること
- レビュー等で見つかった既存（Pre Existent）の問題はタスクのスコープ外でも修正すること（ボーイスカウトルール）
    - ただし、影響範囲が大きかったり、重要な設計判断が必要な問題については別タスクとすることも検討すること
- 規模が小さいため、後方互換性の破壊はためらわず、最も堅牢で合理的、シンプル（NOT EASY）なコードを記述しコードベースをきれいに維持すること

## [MUST] プロダクトコード

- 外部ライブラリは以下の理由からlibrary-contractパターンを使ってcrates/rip-adaptersに切り離し層をつくる
    - 外部ライブラリのバージョンアップ時の挙動確認テスト（契約テスト）を行いやすくする
    - 外部ライブラリを差し替え可能にすることでメンテナンスされなくなった場合の移行を簡単にする
    - サプライチェーン攻撃の影響範囲をアダプター層に限定し、コアドメインロジックを保護する
    - テスト時にFake/Mock実装へ差し替えて高速なユニットテストを可能にする
    - コアドメインロジックを外部ライブラリのAPI破壊的変更から隔離する

## [MUST] テストコード

- ユニットテスト：外部依存なしの高速テスト。ローカルマシンとGithub Actions上のCIで実行される
- 統合テスト：複数のモジュールの連携を確認するテスト。ローカルマシンとGithub Actions上のCIで実行される
- TDD 的アプローチ: 先にテストを書き、それを満たすプロダクトコードを書く
- ユニットテスト、統合テストがリグレッションテストであるとともに **仕様書となる** ように記述する
- モックは最小限に留める。本当にどうしても必要な場合のみシンプルなものに限る
- 各モジュールのテストカバレッジは可能な限り高く保つ

## [MUST] タスク管理の使い分け

GitHub IssuesとClaude CodeのTasksを使いタスクを管理する。

以下の場合にTasksを使用する：

1. **3ステップ以上の実装** - 細かい進捗をユーザーに可視化
2. **サブエージェント起動時** - 並行作業の進捗共有に必須
3. **ユーザーから複数の依頼** - 「AとBとCをやって」のような場合

使用しない場合：
- 単純な1-2ステップの修正
- 調査・質問応答のみのセッション

## [MUST] GitHub Issue 起票ルール

開発中に発見したバグ、改善点、設計上の懸念は GitHub Issue に起票する。

### バグ報告

```
## 現象
何が起きているか

## 再現手順
1. ...
2. ...

## 期待する動作
本来どうあるべきか
```

### 機能・改善

```
## 概要
何をするか（1〜2文）

## 対象ファイル
- `path/to/file.rs` — 変更内容

## 依存
- 前提となる Issue があればリンク
```

### 共通ルール

- タイトルは簡潔に内容がわかるようにする
- ラベルを適切に付与する（bug, enhancement 等）
- 影響範囲が大きい場合はタイトルに対象コンポーネントを含める
