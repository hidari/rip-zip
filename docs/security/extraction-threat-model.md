# ZIP展開時のセキュリティ脅威モデル

## 概要

このドキュメントは、将来のZIP展開機能の実装に先立ち、想定されるセキュリティ脅威と対策方針を定義する。
ZIP生成機能で確立した多層防御パターンを展開機能にも適用し、設計段階からセキュリティを組み込む。

## 脅威一覧

### 1. Zip Bomb（圧縮爆弾）

圧縮率が極端に高いアーカイブを展開させ、ディスク容量やメモリを枯渇させるDoS攻撃。

攻撃シナリオ:
- 数KBのZIPファイルが展開時に数GB〜数TBに膨張する
- ネストされたZIP内ZIPで再帰的に膨張する（再帰的zip bomb）
- 多数の同一ファイルエントリで展開ファイル数を爆発させる

対策方針:
- 展開先の合計サイズ上限を設定する（既存 `MAX_TOTAL_SIZE` を再利用）
- 圧縮比率チェック: 各エントリの `compressed_size / uncompressed_size` が閾値を超える場合にスキップ
- 展開ファイル数の上限を設定する（既存 `MAX_FILE_COUNT` を再利用）
- 個別ファイルサイズの上限を設定する（既存 `MAX_FILE_SIZE` を再利用）
- ネストされたアーカイブの自動展開は行わない（スコープ外）

### 2. ZipSlip（パストラバーサル）

ZIPエントリ名に `../` を含めることで、展開先ディレクトリの外にファイルを書き込む攻撃。

攻撃シナリオ:
- エントリ名 `../../../etc/cron.d/malicious` で任意のディレクトリにファイルを配置
- バックスラッシュ `..\..\windows\system32\evil.dll` でWindows環境を攻撃
- 不可視Unicode文字を混入した `.\u{200B}./etc/passwd` でチェックを回避

対策方針:
- 既存 `has_path_traversal()` でエントリパスを検証（`Path::components()` による `ParentDir` 検出）
- 既存 `sanitize_zip_entry_path()` で各パスセグメントをサニタイズ
- 展開先パスが対象ディレクトリ配下であることを `canonicalize()` で最終確認
- バリデーション順序: traversal check → sanitize → canonicalize確認

### 3. Symlink攻撃

ZIPエントリにシンボリックリンクを含め、展開先のファイルシステムを操作する攻撃。

攻撃シナリオ:
- ZIP内のsymlinkエントリで `/etc/shadow` へのリンクを作成
- 先にsymlinkを展開させ、後続のファイル書き込みでリンク先を上書き
- 既存ディレクトリへのsymlinkで書き込み先をリダイレクト

対策方針:
- ZIPエントリのsymlinkフラグを検出しスキップする（生成時と同様の方針）
- 展開先パスの各コンポーネントがsymlinkでないことを検証する
- symlinkの展開を許可するオプションは提供しない（攻撃面の最小化）

### 4. ファイル上書き

展開先に既存ファイルが存在する場合、意図しない上書きが発生する。

攻撃シナリオ:
- 重要な設定ファイル（`.bashrc`, `.ssh/authorized_keys`）を上書き
- 同一ZIP内に同名エントリが複数存在し、後のエントリで上書き

対策方針:
- デフォルトで既存ファイルを上書きしない（スキップまたはエラー）
- `--overwrite` フラグによる明示的な許可を要求
- ZIP内の重複エントリ名を検出し、警告を出力

### 5. 危険なUnixパーミッション

ZIPエントリに格納されたパーミッションが、展開後にセキュリティリスクとなる。

攻撃シナリオ:
- setuid/setgidビットが設定されたバイナリの配置
- 全ユーザー書き込み可能（0777）なディレクトリの作成
- 実行権限付きスクリプトの配置

対策方針:
- setuid（0o4000）/ setgid（0o2000）/ sticky（0o1000）ビットを除去
- パーミッションの上限をマスクする（例: ファイルは最大0o755、ディレクトリは最大0o755）
- パーミッション変更時にイベントを通知（既存 `ZipEvent` パターンを拡張）

### 6. 不正なファイル名

ZIPエントリ名に不可視文字、制御文字、Windows予約名などが含まれる。

攻撃シナリオ:
- RTL制御文字でファイル名を偽装（`invoice\u{202E}fdp.exe` → 表示上 `invoiceexe.pdf`）
- Windows予約デバイス名（CON, NUL等）でシステムに影響
- ゼロ幅文字でファイル名を見えなくする

対策方針:
- 既存 `sanitize_zip_entry_path()` を展開パスに適用
- 不可視Unicode除去、危険文字置換、Windows予約名エスケープの全レイヤーを再利用
- サニタイズ時に `ZipEvent::PathSanitized` イベントを通知

### 7. リソース枯渇

大量のエントリやネスト構造でCPU/メモリ/ディスクI/Oを枯渇させる。

攻撃シナリオ:
- 数十万のファイルエントリでファイルシステムのinode枯渇
- 深いディレクトリ構造でパス長制限に到達
- 極端に長いファイル名でメモリ消費

対策方針:
- ファイル数上限（既存 `MAX_FILE_COUNT`）を適用
- ファイル名長チェック（既存 `is_filename_too_long()`）を適用
- 展開時のディレクトリ深度は作成するディレクトリ数で間接的に制限される

## 既存コンポーネントの再利用マップ

| 既存コンポーネント | ファイル | 展開での用途 |
|---|---|---|
| `has_path_traversal()` | `validation.rs` | ZIPエントリパスのtraversal検出 |
| `sanitize_zip_entry_path()` | `path_utils.rs` | 展開先パスのサニタイズ |
| `sanitize_filename_core()` | `path_utils.rs` | 個別セグメントのサニタイズ |
| `normalize_path_separator()` | `validation.rs` | エントリ名のパス区切り正規化 |
| `is_filename_too_long()` | `validation.rs` | ファイル名長チェック |
| `should_skip_large_file()` | `validation.rs` | 個別ファイルサイズチェック |
| `check_total_size()` | `validation.rs` | 合計サイズチェック |
| `check_file_count()` | `validation.rs` | ファイル数チェック |
| `MAX_FILE_SIZE` 等 | `config.rs` | サイズ制限定数 |
| `FileSkipReason` | `types.rs` | スキップ理由の型安全な表現（バリアント追加） |
| `ZipEvent` | `types.rs` | イベント通知パターン（バリアント追加） |
| `ZipError` | `error.rs` | エラーハンドリング |

## 展開フロー案

```
1. 対象ZIPファイルのバリデーション
   - ファイル存在・読み取り権限チェック

2. エントリ一覧の事前スキャン
   - 合計展開サイズの概算チェック（zip bomb早期検出）
   - 合計ファイル数チェック
   - 重複エントリ名の検出

3. 各エントリの展開ループ
   a. パストラバーサル検出 → スキップ
   b. パス区切り正規化
   c. パスセグメントサニタイズ
   d. ファイル名長チェック → スキップ
   e. 個別ファイルサイズチェック → スキップ
   f. 圧縮比率チェック → スキップ
   g. symlinkエントリ検出 → スキップ
   h. 展開先パスのcanonicalizeチェック（対象ディレクトリ外への書き込み防止）
   i. 既存ファイル上書きチェック
   j. パーミッション検証・マスク適用
   k. ファイル展開
   l. 累計サイズ更新・チェック
   m. イベント通知

4. 完了処理
   - 展開統計の出力
```
